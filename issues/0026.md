---
title: "Filesystem tools should accept optional named mount parameter"
status: open
priority: medium
---

## Context

The `bash` tool accepts an optional `mount` parameter that lets the agent target a specific named mount (and its associated runner). However, the filesystem tools — `read`, `write`, `edit`, `ls`, `find`, `grep` — do not. They always resolve paths against the primary mount via `resolvePathViaMountOrCwd(path, cwdProvider, mountResolver)` which calls `mountResolver(nil)`.

This means in a multi-mount, multi-runner setup, the agent cannot directly read or write files on a non-primary mount without first switching the primary. That's unworkable for true multi-computer use — e.g., an agent with "frontend" on a local Mac and "backend" on a remote Linux runner can't `read(path: "src/main.rs", mount: "backend")`.

## Tasks

- Add an optional `mount` string parameter to `read`, `write`, `edit`, `ls`, `find`, `grep` tool schemas
- Thread the mount name through `resolvePathViaMountOrCwd` (or replace with a direct `mountResolver(params.mount)` call)
- Ensure the resolved runner is used for the actual filesystem operation
- Update tests
